    🔧 故障排除

    收集了最常见的部署和使用问题及其解决方案， 帮助快速定位和解决问题。

    环境配置问题

    问题：Elasticsearch 启动失败

    ❌

    错误信息

    max virtual memory areas vm.max_map_count [65530] is too low

    解决方案：

    bash 
    # Linux（临时）
sudo sysctl -w vm.max_map_count=262144

# Linux（永久）
echo "vm.max_map_count=262144" | sudo tee -a /etc/sysctl.conf
sudo sysctl -p

# Windows WSL2（在 WSL2 中执行）
sudo sh -c 'echo "vm.max_map_count=262144" >> /etc/sysctl.conf'
sudo sysctl -p

    问题：Ollama 连接失败

    ❌

    错误信息

    Connection refused: http://localhost:11434

    排查步骤：

    bash 
    # 1. 检查 Ollama 是否运行
ollama list
curl http://localhost:11434/api/tags

# 2. 如果从 Docker 容器访问宿主机 Ollama
# 修改 backend/.env，将 OLLAMA_BASE_URL 改为：
OLLAMA_BASE_URL=http://host.docker.internal:11434  # Windows/Mac
OLLAMA_BASE_URL=http://172.17.0.1:11434            # Linux Docker 默认网关

# 3. 确认模型已下载
ollama pull qwen2.5:7b

    问题：Node.js 版本过低

    ❌

    错误信息

    The engine "node" is incompatible with this module

    需要 Node.js 18.0+。使用 nvm 管理版本：

    bash 
    nvm install 20
nvm use 20
node --version  # 应显示 v20.x.x

    网络连接问题

    问题：前端无法连接 LangGraph Backend（422 错误）

    ❌

    错误信息

    POST /api/langgraph/threads 422 Unprocessable Entity

    常见原因：

    • 请求体格式不匹配（assistant_id 错误）
    • LangGraph Server 版本不兼容
    • NEXT_PUBLIC_AGENT_API_URL 环境变量未设置
    bash 
    # 验证 Backend 可访问
curl http://localhost:2024/health

# 验证 Assistant ID 存在
curl http://localhost:2024/assistants | python3 -m json.tool

# 检查前端环境变量
cat frontend/agent-chat-ui/.env | grep AGENT_API_URL

    问题：SSE 流式响应断开

    可能原因：

    • Nginx 代理超时（proxy_read_timeout 设置不足）
    • 网络中间件（负载均衡、CDN）切断长连接

    解决：确保 Nginx 配置包含 proxy_read_timeout 86400;（24小时）。

    问题：RAG 服务返回 401

    bash 
    # 检查 RAG API Token 配置
# 在 rag/.env 中确认 SECRET_KEY 已设置
grep SECRET_KEY rag/.env

# 通过前端登录 RAG（获取 Token）
curl -X POST http://localhost:9380/v1/user/login \
  -H "Content-Type: application/json" \
  -d '{"email": "admin@ragflow.io", "password": "admin123"}'

    Agent 问题

    问题：LLM 无法识别任务类型

    后装 Agent 使用 LLM 解析用户意图，如果 LLM 能力不足（模型太小）可能无法正确识别。

    解决方案：

    • 升级为更大的模型：OLLAMA_MODEL=qwen2.5:14b
    • 在用户输入中明确指定任务类型，例如："请帮我训练 DMS2.0-人眼 任务"
    • 降低 temperature：OLLAMA_TEMPERATURE=0.2（减少随机性）

    问题：人在回路中断后无法恢复

    前端点击确认后没有反应，流式响应未恢复。

    排查：

    • 检查 thread_id 是否与中断时的请求一致
    • 确认 command.resume 的格式与后端 interrupt() 期望的格式匹配
    • 查看 Backend 日志是否有异常

    问题：商用车 QA 未返回知识库引用

    开启"知识库记忆"后，回答来源于 Ollama 而非 RAG。

    排查步骤：

    1. 确认前端 knowledge_memory: true 已传入请求 context
    2. 检查 motovis_dialog_id 是否在 configurable 中正确传递
    3. 验证 Dialog ID 对应的 RAG Dialog 是否存在且已关联知识库
    4. 检查 RAG 服务是否正常：curl http://localhost:9380/v1/system/status

    性能优化问题

    问题：LLM 响应速度慢

    • GPU 加速：确保 Ollama 使用 GPU（ollama ps 查看是否有 GPU 字样）
    • 模型量化：使用 :q4_0 量化版本（速度提升但精度略降）
    • 并发限制：减少同时进行的对话数量
    bash 
    # 检查 Ollama GPU 使用情况
ollama ps
nvidia-smi  # NVIDIA GPU

# 使用量化模型
ollama pull qwen2.5:7b-q4_0  # 4-bit 量化，更快
OLLAMA_MODEL=qwen2.5:7b-q4_0

    问题：Elasticsearch 内存不足

    ES 默认 JVM 堆内存 512MB（-Xms512m -Xmx512m），如果服务器内存充裕可以调大：

    yaml 
    # docker-compose.yml 中修改
elasticsearch:
  environment:
    - "ES_JAVA_OPTS=-Xms1g -Xmx1g"  # 调整为 1GB

    • 打开浏览器 DevTools → Network 面板，过滤 /api/langgraph 请求，查看 SSE 事件流
    • Console 中查看 Stream error 日志
    • 在 providers/Stream.tsx 的 onError 回调中添加详细日志
